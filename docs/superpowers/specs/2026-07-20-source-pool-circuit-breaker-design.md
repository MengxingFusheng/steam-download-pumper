# 下载源池扩充与自动熔断设计

## 目标

- 扩充经过实际部署服务器验证的中国境内 IPv4 公共 HTTP/HTTPS 下载源。
- 失效源不再持续占用 worker；临时故障恢复后可自动重新加入。
- iKuai 单线路镜像和多 IP 镜像共用同一套下载引擎、配置和源健康逻辑。
- 保持每条线路最多 12 个连接，不通过无限增加线程掩盖源质量问题。

## 现状与根因

当前默认源池包含 4 个测速节点，其中两个节点已经无法稳定连接。Go 下载引擎虽然有共享失败计数和最长约 32 秒的指数退避，但冷却到期后多个 worker 仍会持续重试坏源。四条线路上的 worker 因超时同时等待时，吞吐会周期性降为零。

从部署服务器临时移除两个坏源后，目标带宽为 1600 Mbps 时，60 秒平均吞吐恢复到约 1570 Mbps，说明主机资源、容器连接上限和 iKuai 一对一分流不是当前主要瓶颈。

## 方案选择

采用共享熔断器，而不是永久删除或固定冷却：

- 永久删除无法应对公共节点的临时维护和网络抖动。
- 固定冷却会让多个 worker 在同一时刻重新冲击故障源。
- 共享熔断器可以限制故障请求，并通过单探针安全恢复源。

## 源准入与默认池

候选源必须在实际部署服务器上完成以下验证后才能进入默认池：

1. 域名可解析到 IPv4 地址，并能强制使用 IPv4 建立连接。
2. HTTP 状态正常，响应体可持续读取，不依赖登录、Cookie 或私有凭据。
3. 连续测试期间没有频繁超时或断流，并具有实际下载吞吐。
4. 服务位于中国境内网络，属于公开测速、公开镜像或官方分发资源。
5. 不将同一后端地址的多个域名当作独立容量来源。

默认源池只保存通过验证的 URL。运行时熔断负责处理后续失效；用户在控制台添加的源使用相同健康逻辑。

## 熔断状态机

每个逻辑线路的下载引擎维护一份由全部 worker 共享的源状态。状态包括：

- `healthy`：可被任意 worker 选择。
- `degraded`：出现不足 3 次连续失败，使用 1 秒、2 秒的短退避。
- `quarantined`：连续失败 3 次后隔离，所有普通 worker 跳过。
- `probing`：隔离到期后只允许一个 worker 执行恢复探测。

隔离时间依次为 10 分钟、30 分钟和 60 分钟，之后失败继续保持 60 分钟。任意一次完整、有效的下载成功都会清除连续失败、隔离级别和最近错误，使源返回 `healthy`。

当所有源都不可用时，worker 等待最早的下一次探测时间，并附加少量随机抖动。不得忙循环，也不得同时发起恢复探测。

## 下载选择与事件接口

worker 在健康源之间轮转，继续使用长连接式循环下载并把响应写入 `io.Discard`。熔断器只改变源可用性判断，不改变每线连接扩缩容策略和 12 连接硬上限。

Go helper 的 `source` 状态事件增加以下字段：

- `state`
- `consecutive_failures`
- `retry_after`
- `retry_in_seconds`
- `error`
- `recovered`

Python 控制器保存每条线路最新源状态，并在 `GET /api/sources` 中聚合展示 URL、解析 IPv4、总体健康状态、连续失败数、隔离剩余时间、最近错误和各线路状态。现有字段继续保留，避免破坏当前控制台。

## 两镜像同步

以下内容必须保持共享，不做镜像分叉：

- `cmd/discarder` 下载及熔断实现。
- `steam_pumper/config.py` 默认源池和校验。
- `steam_pumper/engine.py` 状态事件解析。
- `steam_pumper/controller.py` 源健康 API。
- 对应测试。

iKuai 单线路版只运行一个逻辑线路；多 IP 版为每个绑定 IP 创建一个逻辑线路。两者仅拓扑不同，熔断行为完全一致。

## 错误处理

- DNS、连接、TLS、非 2xx 状态、读取超时和响应中断都计为失败。
- 上下文取消和容器正常停止不计入源失败。
- 成功读取响应后才恢复源；空响应或异常短响应不得误判为恢复。
- 日志使用无 ANSI 颜色的 UTF-8 单行文本，防止 iKuai 日志页面出现乱码。

## 测试与验收

### 自动测试

- Go 单元测试覆盖失败阈值、三个隔离等级、单探针竞争、恢复清零和全源隔离等待。
- Go 集成测试用本地健康源和故障源验证故障请求数受限、健康源持续产生流量。
- Python 测试覆盖新增事件解析、API 聚合以及两个拓扑的行为一致性。
- 镜像测试确认两个 Dockerfile 使用同一 helper，且每线最大连接数仍为 12。

### 部署验收

- 同时发布 iKuai 单线路版和多 IP 版的新标签。
- 当前四线服务器更新后连续观察至少 10 分钟。
- 目标 1600 Mbps 时，稳定阶段 60 秒均值达到目标的 90% 以上。
- 人工加入一个故障 URL 后，该源在 3 次连续失败后进入隔离，健康源继续下载，不再出现所有线路同步归零。
- API 能看到隔离状态和剩余时间；故障源恢复后单探针成功并自动重新启用。

## 发布与回滚

发布脚本必须原子构建并推送两个镜像。部署前保留当前镜像 digest 和配置备份；若新镜像无法满足验收，恢复原 digest，并保留已经验证健康的两源配置。
